## 0. 設計思想

目的は情報を保存することではなく、**日々の経験・学習・思考を構造化し、将来の意思決定と事業（コンサルティング）に再利用できる知識資産へ育てること**。

インプット窓口は1つ。すべての音声・URL・テキスト入力を、性質の異なる2つの出力構造に分解する。

- **マインド（Mind Graph）**：思想・価値観・性格特性・仮説など、多軸で自由に関連し合う知識。テーマ横断こそが価値。
- **スキル・アセット（Skill/Asset Tree）**：ツール活用法・事業運営スキル・専門知識など、体系的に積み上がる資産。将来のブログ・コンサル資産の種。

この2つは同じ入力から同時に抽出されることもある（例：ジャーナル1件からマインドの気づきと、ツールの学びの両方が出る）。

## 0.5 保存構造の設計原則（2026-07-19 確定）

このツールの大きな利点は、**リレーショナルDBを持たず、ファイル置き場（Google Drive等）さえあれば動く**ことにある。この利点を活かすため、以下を設計原則として確定する。

**原則：Note・Knowledge・Task・Great Mind・Habitは、すべて1オブジェクト＝1ファイルの独立ファイルとして保存する。**

- Note（Journal/Source/Insight/Inbox）は、その日その場でインプットされた情報の「入れ物」であり、それ自体も1ファイル
- そこから抽出されたKnowledge・Task・Great Mindは、Noteのfrontmatter内に配列として埋め込むのではなく、**専用フォルダの中に、それぞれ独立したファイルとして保存する**
- Habitも同様に独立ファイル。ただしTask同様、ジャーナルや記事から良い習慣を発見して登録することがあるため、`related_notes`で発見元Noteを辿れるようにする

**なぜこの方式にしたか（一度、逆方向で失敗した経緯）**：
2026-07-19、一時的に「Great MindはUI上の専用タブが不要だから、ファイルとしても軽量化してNoteに埋め込んでよい」と判断したが、これは誤りだった。**「UI上どれくらい頻繁に見るか」と「ファイルとしてどう保存するか」は無関係の話**であり、埋め込み方式は以下の代償を伴っていた。
- 検索性の低下：ファイル一覧だけでは中身が分からず、全Noteを開いてYAMLを解析しないと集計できない
- 参照の脆さ：埋め込み方式ではIDを持たせにくく、タイトル文字列で参照すると、タイトルを1文字直すだけでリンクが切れる

独立ファイル化により、この2つの代償を解消した。UI上の見せ方（専用タブを持つか、他画面から時々参照する程度か）は、保存方式とは別レイヤーの話として、都度決める。

**各オブジェクトの「出典・親子関係」の持たせ方**：
- 子オブジェクト（Knowledge/Task/Great Mind/Habit）は`origin_note`フィールドで、生まれたNoteのidを持つ
- 親であるNote側にも、`knowledge_refs` / `task_refs` / `great_mind_refs`という**IDだけの軽量な配列**を持たせ、Noteを開けば何が生まれたか一覧できるようにする（内容の二重管理はしない。実体は子オブジェクト側のファイルのみ）

**フォルダ構成（更新後）**
```
00_Inbox/        未整理・クイックメモ
01_Journal/      日々のジャーナル
02_Sources/      外部情報（Video/Article/Book/Work）
03_Insights/     パターン・実験ログ
04_MOC/          テーマ別ハブ（4テーマ）
05_Reference/    スキル・アセットツリー
06_GreatMind/    賢者の教え（独立ファイル、1ルール1ファイル）
07_Task/         タスク（独立ファイル、1タスク1ファイル）
08_Knowledge/    ナレッジ（独立ファイル、1件1ファイル）
09_Habit/        習慣（独立ファイル、1件1ファイル）
Templates/
```

## 1. インプット処理の優先順位

Knowledge生成時、以下の優先順位で扱う。

1. **Reflection**（ユーザー自身の感想・解釈・経験）— 最重要
2. **Source Summary**（記事・動画の要約）
3. **元ソース**（URL本文・動画内容そのもの）

Reflectionに根拠が薄い場合はconfidenceを下げ、必要に応じてResearch Taskを生成する。

### URL取得ルール
- 記事：本文取得を試みる。成功時はTitle/URL/詳細要約を生成。失敗時は「本文を取得できませんでした。Source Summaryを入力してください」と返す
- YouTube：動画内容の取得は保証されない。取得できない場合、要約記事等の代替ソースを探す。それでも得られない場合はSource Summaryの入力を求める

## 2. マインドグラフ（Mind Graph）仕様

### 2.1 テーマ（Territory）— 4分類、固定
- Healthcare（健康）
- Business（事業）
- Mind（マインド：考え方・価値観・性格特性）
- Relationships（人間関係）

1つのアトムが複数テーマに属してよい（例：睡眠は healthcare × business）。

### 2.2 カテゴリの枝（Branch）
テーマの下に、内容に応じたカテゴリを持つ。例：
- Healthcare：睡眠／食事／運動／日光浴・スキンケア
- Business：戦略
- Mind：マインドセット／性格特性／学習法

**マインドセットと性格特性は固定区分ではなくスペクトラム**。状況依存の考え方（マインドセット）が繰り返されて性格特性に「昇格」したり、逆に性格特性だと思っていたものが状況次第の柔らかいマインドセットへ「降格」したりする。今後、この移行を表現できるようにする。

新しいカテゴリが必要そうだと感じたら、Claudeから「これは新しい枝になりそうです」と提案する。ユーザーの承認を得てから追加する。

### 2.3 知識アトム（Knowledge Atom）
**ルール**
- 1ノード1主張
- タイトルは結論形式（例：「小規模企業共済は高所得者ほど節税効果が大きい」のような言い切り文）
- 定義だけのノードは作らない。再利用可能な内容のみ抽出
- 事実だけでなく、経験から得た原則・仮説・将来検証したい考えもアトムとして保存する
- **Knowledgeは一度書いたら完成ではない。Noteと違い、複数のNoteから知見が積み重なるたびに育っていく存在**（詳細は2.3.1）

**スキーマ（独立ファイル：`08_Knowledge/{id}.md`。2026-07-19より、Noteへの埋め込みではなく独立ファイルに変更）**

| フィールド | 内容 |
|---|---|
| title | 結論形式の短いタイトル（グラフ表示用） |
| summary | 現時点での最良の理解（詳細表示用）。1つのNoteだけの解釈ではなく、`evidence`に積み上がった根拠を踏まえて書き直されていく本体 |
| evidence | 背景・根拠を積み上げていくログ（削除せず追記。詳細は2.3.1） |
| themes | 所属テーマ（複数可） |
| branch | 所属カテゴリ |
| confidence | 信頼度 0〜100 |
| source_quality | High / Medium / Low（複数evidenceがある場合は最も質の高いものを採用） |
| created | 最初にこのKnowledgeが生まれた日 |
| updated | 直近でevidenceが追加された日 |
| origin_note | 最初にこのKnowledgeを生んだNoteのid（0.5節） |
| related_notes | それ以降、evidenceとして関わった関連ノート |
| revises | （該当する場合）どのKnowledgeの考えを更新したか |
| merged_into | （該当する場合）このKnowledgeが統合された先の正規Knowledge id。統合後は新規evidenceを持たず、参照専用になる |

**注記**：「Knowledge」と「アトム」は同じ概念を指す（設計初期はアトムと呼んでいたが、Knowledge Builder参考時にKnowledgeという呼称も混在した。以後はKnowledgeに統一する）。titleはグラフ上で表示する短い結論文、summaryはタップして開いた時に読む「なぜ・どうなった」を含む本体。親Note側には`knowledge_refs: [id, ...]`という軽量な参照リストのみを持たせる。

**confidenceの目安**
- 90〜100：高品質ソースで裏付け済み
- 70〜89：比較的信頼できる
- 40〜69：一部推測を含む
- 0〜39：主観・仮説中心

**source_qualityの目安**
- High：学術論文／公的機関／専門書
- Medium：企業ブログ／一般書籍／YouTube
- Low：個人的体験／感想／個人ブログ

### 2.3.1 Knowledgeは育つ ― summary と evidence の役割分担

**背景にある考え方**：URLの解釈や1回のジャーナルだけから生まれたKnowledgeは、まだ「質の高いナレッジ」とは言えない。いろいろな知見が複数のNoteをまたいで折り重なることで、初めて見返す価値のあるナレッジに育っていく。NoteはURL・動画の要約など、その時点の入力を丸ごと保存する「一次資料のアーカイブ」であり、リッチだが原則書き換えない。一方Knowledgeは、Noteほどのボリュームは要らないが、**時間をかけて深掘りされ、太っていく**べき存在。

**`evidence`のフォーマット**（追記型ログ）
```yaml
evidence:
  - date: "2026-07-02"
    source: "20260702-source-01"
    note: "動画で推奨されていた23時就寝と一致することに気づいた"
  - date: "2026-07-02"
    source: "20260702-journal-01"
    note: "実際に60日間継続できており、以前より睡眠の質・日中パフォーマンスが向上したと感じている"
```

**運用ルール**
1. 似たKnowledgeが新しいNoteから見つかったら、新規Knowledgeを作らず、既存Knowledgeの`evidence`に1件追記する（2.6の自動強化と連動）
2. `evidence`が増えるたびに、Claudeは`summary`を**書き直す**（積み上げるのはevidenceだけ。summaryは常に「今のところの最良の要約」を保つ）
3. `evidence`が1件しかない（＝1つのNoteの解釈だけに基づく）Knowledgeは、まだ育っていない状態。confidenceが低い場合は特に、Research Task「この知見をさらに深掘りする」を自動生成し、深掘りを促す
4. Note詳細画面・Knowledge詳細画面のどちらでも、`evidence`は時系列のタイムラインとして表示する

### 2.4 Summary Knowledge
1つの入力（ソース）ごとに、最も重要な学び・原則・価値観を1つ抽出し、その入力から生まれた複数のKnowledgeを束ねる上位ノードとして作成する。

### 2.5 リンクの種類
| 種類 | 意味 | 表現 |
|---|---|---|
| hierarchy | テーマ→枝→Knowledgeの階層 | 実線・グレー |
| cross | テーマを横断する関連性 | 破線・紫 |
| revises | 過去のKnowledgeの考えを更新した | 矢印付き実線・オレンジ |
| source | Knowledgeの抽出元 | 点線・グレー |
| experiment | 同じ行動変化に紐づく複数の観測結果 | 破線・紫（太め） |

### 2.6 出現頻度による自動強化
似たような（完全に同一でなくてよい）知識が複数ソースから得られた場合、新規Knowledgeを作らず既存Knowledgeへ統合・強化する。その際`evidence`に1件追記し、confidence・関連ソース数を更新し、`summary`を書き直し、リンクを太く／ノードを大きく表示する。

**この自動強化とは別に**、複数ノートで構造が同型の葛藤・パターンが独立に出現した場合の「Insightノートへの昇格」は、回数の閾値ではなく、Claudeが「明らかに強い関心・繰り返しの重み」を感じたときに手動で提案する。提案時は判断理由を必ず添える。

### 2.7 実験ログ（Experiment Log）
同じ生活習慣・行動の変化について、複数の異なる症状／観測結果が時間をかけて蓄積される場合、単発のKnowledgeではなく「実験ログ」として追記型のノードに育てる（例：夕食抜き実験ログ）。日付ごとに観測を積み上げる。仕組みとしては2.3.1のevidenceログと同型（実験ログは複数の異なる症状を横断的に束ねる点が異なる）。

### 2.8 Great Mind（賢者の教え・背景情報）

**位置づけの経緯（2026-07-19、2回の見直しを経て確定）**：
1. 2026-07-13時点：専用タブは不要だが、独立ファイル（`06_GreatMind/`）としては残す
2. 同日中に一度：「専用タブが不要＝ファイルも軽量化すべき」と誤って判断し、Note内埋め込み方式に変更
3. **最終的に②を撤回し、①に戻した**：保存構造（独立ファイルにするか）とUI上の見せ方（専用タブを持つか）は無関係の話だと整理し直したため（詳細は0.5節）

- **`06_GreatMind/`フォルダに、1ルール＝1ファイルとして保存する**（独立ファイル・独自の`id`を持つ）
- `origin_note`フィールドで、その教えが最初に登場したNote（記事・動画のSource Note、または他に出典がなければJournal）のidを持つ。従属する側のNoteには、`great_mind_refs`という軽量な参照リストを持たせる
- ユーザー自身の解釈・当てはまり度合いは、これまで通りGreat Mind側には持たせず、**それを引用するKnowledge側の`alignment`フィールド**にのみ記録する（この方針は一貫して変更なし）
- Great Mindを一覧として見返す専用画面は、UI上には（あっても）ほぼ使わない想定。「たまに参照する背景情報」として、メンターコメント（2.9）が主な露出経路になる

**スキーマ**
```yaml
id: "（例：20260706-source-02-gm-01）"
schema_version: 1
type: great_mind
title: "（1つの行動指針・ルール。「〇〇する」という習慣文）"
sage: "（賢者名）"
sources:
  - title: "（動画・記事タイトル）"
    url: "（URL）"
rationale: "（なぜ重要とされているか）"
precondition: "（有効になる前提条件）"
pinned: true | false
origin_note: "（生まれたNoteのid）"
visibility: private
```
出典が明確でない場合（伝聞・要約からの抽出等）は、`source_note:`という一言フィールドで由来を書き添えてよい（例：「〇〇さんが△△と語った内容から」）。

### 2.9 鏡・メンターとしてのフィードバック

**コンセプト**：このツールは「自分を映し出す鏡であり、メンターである」という位置づけ。alignmentの生データをテーブルで見せるのではなく、**温かい一言コメント**として合成し、アプリを開いた時に自然に目に入るようにする。

例：「昨日のジャーナルは『最重要な1つに時間と労力を集中させる』というマインドセットを体現していていいですね！」

**仕組み（案）**：
1. 直近のJournal/KnowledgeのうちalignmentがついているものからPickする
2. stanceが`agree`のものを優先的に肯定的なコメントとして表示する（`disagree`/`tension`の場合は、否定するのではなく「〇〇については迷いがあるようですね」のような、思考を後押しする言い方にする）
3. 表示場所は①のCapture画面（アプリを開いて最初に見る場所）を想定
4. コメントは1日1〜2件程度に絞り、通知過多にならないようにする

### 2.10 将来構想：Great Mindの共有ライブラリ化
現在はユーザー本人が気に入った賢者のGreat Mindを個人のVaultに登録していく形だが、将来的には**参考文献がしっかりしているGreat Mindに限り、他のアプリ利用者からの登録も受け付け、共有ライブラリとして誰でも選んで使えるようにしたい**という構想がある。

これは技術的には、個人のGoogle Drive/GitHubに閉じたデータではなく、**全ユーザー共有のデータベース（例：Supabase等）が必要**になる、フェーズ2以降の話（前回のインフラロードマップの「20人〜」フェーズに相当）。今は個人のVaultにNote単位で埋め込む形を徹底し、共有化は将来の課題として持ち越す。

### 2.11 習慣（Habit）— 新設・毎日チェックする本命機能

**位置づけ**：Great Mindが「たまに参照する背景の知恵」であるのに対し、Habitは**毎日チェックする、頻繁に触る本命の機能**。ユーザーが本当に管理したかったのはこちら。

**具体例**：「5時半に起きる」「22時に寝る」「1日1回は体を動かす」のような、毎日繰り返す行動。

**朝チェック／夜チェックの選択**：Habitごとに、チェックを促すタイミングが「朝」か「夜」かを設定できる（`check_time`）。例えばウェイトトレーニングを朝やる人・夜やる人がいるため、ユーザーごとに選べるようにする。

**チェックの仕組み**：
- 登録した各Habitについて、`check_time`に応じたタイミングで「今日はできましたか？」とYes/Noで確認する
- UI操作は、Taskの「見直しタイム」と同じ**フリック（スワイプ）＋外側タップ**の仕組みを流用する（左＝できなかった、右＝できた、など）
- 1日1件ずつではなく、その日まだ未回答のHabitをカードスタックのように順番に確認していく

**継続日数（ストリーク）**：
- Yesが連続した日数を「ストリーク」として計算する
- アプリを開いた時（特に朝）に「〇〇日連続達成中」を表示し、モチベーションにする
- カレンダー表示で、日ごとの達成・未達成を一覧できるようにする

**Great Mindとの関係**：
- Habitの実行記録そのものが、間接的にGreat Mindのalignmentのエビデンスにもなりうる（例：「22時に寝る」を継続できていることは、関連するGreat Mindへの`agree`の裏付けになる）。ただしこれは自動連携の将来課題とし、今はHabit自体の記録に専念する

**スキーマ（通常のHabit。独立ファイル：`09_Habit/{id}.md`）**
```yaml
type: habit
id: "..."
schema_version: 1
title: "（習慣。例：1日1回は体を動かす）"
themes: ["healthcare"]
check_time: "morning" | "night"     # 朝チェックか夜チェックか
special: null
related_notes: []                   # ジャーナルや記事でこの習慣を発見した場合、そのNoteのid（Taskと同じ考え方）
created: ...
status: "active" | "archived"
log:
  "2026-07-08": true
  "2026-07-09": false
```

### 2.12 特別な習慣（起床・就寝・コアトレーニング）

起床・就寝・コアトレーニング（ウェイトトレーニング／歩数）は、他のHabitと違い**自動判定またはアプリ内の専用アクションと連動する**、基本的な習慣として特別枠で扱う。

**① 起床（special: "wake"）**
- `target_wake_time`（例：05:30）を設定する
- Capture画面に常設された「☀️ おはよう」ボタンを、設定時刻から**30分以内**にタップすると、その日は自動的にクリア（`log[today] = true`）になる
- 30分を過ぎてから押した場合、または押さなかった場合は未達成

**② 就寝（special: "sleep"）**
- `target_sleep_time`（例：22:00）を設定する
- Capture画面に常設された「🌙 おやすみ」ボタンを、設定時刻の**30分前〜60分前の間**にタップすると、その日はクリアになる（スクリーンタイムを避けるため、その時間帯より前に区切りをつける意図）
- ウィンドウ外にタップした場合は未達成

**③ コアトレーニング（special: "weight_training"）**
- ウェイトトレーニングの実施。`check_time`（朝／夜）はユーザーが選択（朝にやる人・夜にやる人がいるため）
- 判定は通常のHabitと同様、Yes/Noのフリックチェックで行う（自動判定ではない）

**④ 歩数（special: "steps"）**
- 1日の目標歩数（`step_goal`。デフォルト10000）に対する実績を記録する
- Yes/No形式ではなく、**歩数そのものを毎日入力する**数値ログ
- 7日平均・14日平均・28日平均を算出し、長期的に目標歩数に届いているかを確認する
- 直近14日間は、日ごとの実績も一覧できるようにする
- **自動連携（Apple HealthKit / MyFitnessPal API）は、Webアプリ（PWA）の技術的制約により現時点では組み込めない**（HealthKitはネイティブiOSアプリ専用でWebからは一切アクセス不可。MyFitnessPal APIも非公開・新規受付停止中）。当面は手動入力とし、将来的にiOSショートカット経由での間接連携を検討する

## 4. Task（タスク）

Knowledgeとは別の、独立したオブジェクトとして扱う。

### Action Task
現実世界で実行する行動。例：ビジョンマップを作る、社長へ退職を報告する。

### Research Task
Knowledgeの精度向上のための調査タスク。以下の場合に自動生成する。
- confidenceが低い
- 根拠・ソースが不足している
- 仮説中心の内容である

### 4.1 管理方式：期日ベースではなく「優先順位＋完了フラグ＋依存関係」

Taskは基本的に**期日（due date）で管理しない**。代わりに以下の3つで管理する。

1. **優先順位（priority）**：数値が小さいほど優先度が高い。同じテーマ内での相対的な順位づけに使う
2. **完了フラグ（status）**：`open`か`done`かだけを持つ。いつまでにやるかではなく、やったかどうかだけを記録する
3. **依存関係（depends_on）**：そのタスクに着手する前に終わらせておくべき「先行タスク」のIDを配列で持つ。後続タスクは別フィールドを持たず、他のタスクの`depends_on`を検索して逆算する（Great Mindのalignmentと同じ考え方で、関係は片方向にだけ保存し、逆方向は都度計算する）

**期日を完全に禁止するわけではない**：明確に「この日までにやる必要がある」タスク（例：予約の締切）には`due`を任意で残せるが、基本方針はあくまで優先順位ベースとする。

### 4.2 テーマ分類

Taskにも既存の4テーマ分類（Healthcare/Business/Mind/Relationships）を適用する。基本的には親Noteのthemesを継承するが、Task単体で別のthemesを持たせても良い（Knowledgeと同じ考え方）。UI上ではテーマごとにタスク一覧を分けて表示できるようにする。

### 4.3 スキーマ（独立ファイル。`07_Task/{id}.md`）
```yaml
id: "..."
schema_version: 1
type: "action" | "research"
text: "（タスクのタイトル。短い1文）"
body: "（任意。詳細・メモ。ユーザーが自由に編集できる）"
themes: ["business"]              # 親Noteから継承、または個別指定
priority: 1                        # 数値が小さいほど優先度が高い。テーマ内での相対順位
status: "open" | "done" | "someday"  # someday = 見直しの結果、一旦保留にしたタスク（アーカイブ）
depends_on: ["task_id", "..."]     # 先行タスク（同じ07_Task/フォルダ内のid、任意・複数可）
due: "（任意。明確な締切がある場合のみ）"
created: "（作成日）"
last_confirmed: "（直近で見直しが行われた日。nullなら未見直し）"
archived_at: "（statusがsomedayになった日。復活したらnullに戻す）"
origin_note: "（生まれたNoteのid）"
```
親Note側には`task_refs: [id, ...]`という軽量な参照リストのみを持たせる（詳細は0.5節）。

### 4.4 放置タスクの見直しと、保留タスクのアーカイブ・復活

**見直し（stale review）**：`created`（または`last_confirmed`）から一定期間（目安5日）動きがないopenタスクは、「まだ重要か」をユーザーに確認する対象になる。UIではスワイプ（左＝保留／右＝まだ重要）または明示的なボタンで回答する。

- **「まだ重要」**：`last_confirmed`を今日の日付に更新するだけ。優先順位は変えない。次の見直しサイクルまで再度は聞かれない
- **「一旦保留」**：`status`を`someday`に変更し、`archived_at`に日付を記録する。通常のタスク一覧（テーマ別・クロステーマ）からは除外され、アーカイブとして保持される（削除はしない）

**復活（revival）**：新しいJournal/Sourceを処理する際、その内容がアーカイブ（`status: someday`）されているタスクと意味的に近いと判断した場合、そのタスクを復活させる。
1. `status`を`open`に戻す
2. `archived_at`をnullに戻す
3. `last_confirmed`を今日に更新
4. **優先順位を、他のタスクより少しだけ高く設定する**（例：そのテーマの現在の最小priority値より1つ小さい値にする）。これは「何度も気になっている・繰り返し出てくる関心事」であることの表れなので、通常の新規タスクより優先度を上げて扱う

この一連の判断（アーカイブとの類似性の検出、復活の実行）はClaudeが新規入力の処理時に行う。ユーザーからは「前に保留にした件、また気になってるんですね」と自然に見える形になるのが理想。

### 4.4.1 完了操作（2026-07-23 実装）

タスクカードのチェックボックスをタップすると、`status` を `open` ⇔ `done` で切り替える。完了済みタスクは一覧の上位表示から自動的に「もっと見る」アコーディオン内へ退避する（既存 UI の挙動をそのまま利用）。

**Drive 永続化の設計判断（経緯）**：

2026-07-23 時点で、UI（チェックボックス）と状態遷移ロジック（`toggleTaskDone` 関数）は既に実装されていたが、メモリ上の状態変更のみであり Drive への書き出しは行われていなかった（リロードすると元に戻る）。これは「Capture以外からの書き込み」全体が未着手だったことに起因する。

まず最小実装として、Task完了操作のみ Drive 永続化を実装した。

**実装方針**：
- 既存 UI（チェックボックス）をそのまま利用。UI刷新は行わない
- `toggleTaskDone(id)` を async 化
- メモリ上の `status` を即座に更新 → UI 再描画 → バックグラウンドで Drive へ書き込む（楽観的更新）

- `createOrUpdateFile()`（既存、2054行目）を利用。同名ファイルが Drive 上に存在すれば PATCH、なければ POST
- 失敗時は alert で警告。メモリ状態はロールバックせず、リロードまで変更は保持される（リロード後は Drive 上の正しい状態に戻る）
- Drive 未接続時は警告ログのみ。モック配列での動作確認は可能

**未着手の関連操作**（今後の予定）：
- タイトル変更（`saveTaskTitle`）、本文変更（`saveTaskBody`）、Due date 変更（`saveDueDate` / `clearDueDate`）、依存タスク追加・削除（`addDep` / `removeDep`）も同様にメモリ上のみ
- これらも同じパターン（楽観的更新 + Drive 書き戻し）で実装可能

## 4.5 （将来構想）筋トレ管理ツール：ダブルプログレッション式レベル算出

現時点では独立したツールとして別途作る予定だが、算出ロジックだけ先に確定させておく。

**前提**：1種目あたり1日2セット、1セットの上限8回。

**状態**：`base`（現在サイクルの軽い方の重量）、`increment`（1単位の重量）、`level`（1〜8）

**8レベルの反復サイクル**
- レベル1〜4：1セット目＝`base+increment`の重量で5→8回まで伸ばす（2セット目は`base`×8回で固定）
- レベル5〜8：1セット目は`base+increment`×8回で固定。2セット目が`base+increment`の重量で5→8回まで伸ばす
- レベル8達成（両セットとも新重量×8回成功）で次サイクルへ：`base = base + increment`、`level`は1に戻る

このサイクルが重量を上げながら無限に繰り返されるため、**個別に定義すべきレベルは1〜8の8つだけ**でよい。

**表示形式**：`(base + increment)kg レベル(1〜8)`

**週次の目標**：週に1回以上レベルアップできていればOKという基準。サイクルをまたぐと数値が7つ下がって見える（レベル8→次サイクルのレベル1）ため、週次の伸びを正しく計算するには、サイクルをまたいでも下がらない**通算レベルアップ回数（monotonic counter）**を別途保持し、そちらの差分で「今週+Nレベル」を算出する。

## 5. 保存形式

**Markdown + YAML frontmatter** で統一する。Obsidian・自作Webアプリのどちらに進んでも流用できるようにするための決定。

### 5.1 将来を見据えた frontmatter 拡張（重要）

これまでのノートは、本文（プレーンな文章）の中にアトム・Taskを埋め込んでいた。これは人間には読みやすいが、将来スクリプトやDBに移行する際、文章から再度アトムを抽出し直す必要が出てしまう。

そこで、**frontmatterの中に構造化データを直接持たせる**ことで、本文は今まで通り人間向けの読みやすい文章のまま、機械的な読み取りはfrontmatterのYAMLだけで完結するようにする。

**追加するフィールド**

| フィールド | 内容 | 理由 |
|---|---|---|
| `id` | このノート固有の安定ID（例：`20260704-journal-03`） | タイトルやファイル名を変更してもリンク切れしないようにするため。related_notesは表示上ファイル名でよいが、内部的にはidで紐付ける |
| `schema_version` | スキーマのバージョン番号（例：`1`） | 将来スキーマを変更した時、古いノートを自動移行できるようにするため |
| `updated` | 最終更新日 | `created`とは別に持つ。特にReference（追記型）ノートで重要 |
| `visibility` | `private` / `shareable` / `public` | 将来の一般公開を見据え、どのノートを公開・共有してよいか最初から持たせておく。Mind系は原則private、Skill/Asset系はshareable候補が多い |
| `knowledge` | このノートから抽出されたアトムを配列として構造化して持つ（下記フォーマット） | 本文の文章とは別に、機械可読なデータとして持たせる |
| `tasks` | Action Task / Research Taskを配列として構造化して持つ | 同上 |

**knowledge配列のフォーマット**
```yaml
knowledge:
  - id: "20260704-atom-01"
    title: "最優先タスクに過集中し、それが未解決の間は他のタスクに集中できない性格特性を持つ"
    themes: ["mind"]
    branch: "性格特性"
    confidence: 85
    source_quality: "Low"
    revises: null
    tags: ["personality", "focus"]
```

**tasks配列のフォーマット**
```yaml
tasks:
  - id: "20260704-task-01"
    type: "action"
    text: "社長・取締役へ退職を報告する"
    due: "2026-07-18"
    status: "open"
  - id: "20260704-task-02"
    type: "research"
    text: "過集中傾向とどう付き合うか、心理学的アプローチを調べる"
    status: "open"
```

これにより、将来Vaultを丸ごとスクリプトでスキャンして、`knowledge`と`tasks`の配列だけを集めれば、そのままDBやグラフJSONに変換できる。本文の文章を解析する必要は一切なくなる。

### 5.2 フォルダ構成
```
00_Inbox/        未整理の書き起こし
01_Journal/      日々のジャーナル（Morning / Walk）
02_Sources/      外部情報（Video / Article / Book / Work）
03_Insights/     複数ノートから見えたパターン・実験ログ
04_MOC/          テーマ別ハブノート（Healthcare / Business / Mind / Relationships）
05_Reference/    スキル・アセットツリー（Skill / Asset）
06_GreatMind/    目標とする理想の思考・人物像（ユーザーが能動的に登録）
Templates/       各typeのひな形
```

## 6. 1回の入力処理ごとの標準出力フォーマット

新しい入力を処理するたびに、以下の順で整理する。

1. Source Information（Title / URL / 詳細要約 / Reflection）
2. Knowledge（confidence・source_quality付きのアトム）
3. Summary Knowledge
4. Action Task
5. Research Task
6. （任意）設計・システム改善メモ — このセッションで気づいたルール改善案があれば添える

## 7. 明示的に「自動化しない」と決めたポイント

- Insight・実験ログへの昇格判断：固定回数ではなく、Claudeの手動判断＋理由の説明
- スキル・アセットのLv1〜2：ユーザーが決定
- スキル・アセットのLv3の変更：提案ベース、ユーザー確認後に変更

それ以外（似た知識の統合・強化、Lv4以降の再構成、Research Taskの自動生成）は自動でよい。
